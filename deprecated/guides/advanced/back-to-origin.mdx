Back-to-origin rules let OSS automatically fetch objects from an origin server when a requested object does not exist in the bucket. This enables seamless data migration, hot data caching, and fallback access patterns.

## Rule types

| Type | Description |
|------|-------------|
| **Mirror-based** | OSS fetches the object from the origin, stores it in the bucket, and returns it to the client. Subsequent requests are served from OSS. |
| **Redirect-based** | OSS returns an HTTP 302 redirect to the origin URL. The client fetches the object directly from the origin. |

## Configure Back-to-Origin

<Steps>
  <Step title="Create a rule">
    Navigate to your bucket, choose **Basic Settings > Back-to-Origin**, and click **Create Rule**.
  </Step>
  <Step title="Select the rule type">
    Choose **Mirror** for transparent caching or **Redirect** for URL redirection.
  </Step>
  <Step title="Set the origin URL">
    Enter the origin server URL. For mirror-based rules, OSS fetches objects from this URL and stores them locally.
  </Step>
  <Step title="Configure conditions">
    Set the HTTP status code that triggers the rule (typically 404) and optional prefix/suffix filters.
  </Step>
</Steps>

## Mirror-Based Back-to-Origin

When a client requests an object that does not exist:

1. OSS returns a 404 internally and triggers the mirror rule.
2. OSS fetches the object from the origin server.
3. OSS stores the object in the bucket and returns it to the client.
4. Future requests for the same object are served directly from OSS.

<Tip>
Mirror-based back-to-origin is ideal for migrating data from another storage system. As objects are accessed, they are automatically copied to OSS with zero downtime.
</Tip>

## Redirect-Based Back-to-Origin

When a client requests an object that does not exist:

1. OSS returns an HTTP 302 redirect pointing to the origin URL.
2. The client follows the redirect and fetches the object from the origin.

## Use cases

- **Migration**: Gradually migrate data from an on-premises server or another cloud provider without copying everything upfront.
- **Hot caching**: Cache frequently accessed objects in OSS while the full dataset remains on the origin.
- **Fallback**: Serve content from a backup origin when objects are missing.
