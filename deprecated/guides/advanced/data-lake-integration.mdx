OSS serves as a scalable, cost-effective storage layer for data lake architectures. Integrate OSS with analytics engines like MaxCompute, E-MapReduce, Data Lake Analytics, and open-source tools to query data in place without moving it.

## Integration services

| Service | Description |
|---------|-------------|
| **MaxCompute** | Serverless big data processing. Access OSS data as external tables. |
| **E-MapReduce (EMR)** | Managed Hadoop/Spark clusters using OSS as the primary storage layer (HDFS replacement). |
| **Data Lake Analytics (DLA)** | Serverless SQL queries directly on OSS data in CSV, JSON, Parquet, and ORC formats. |
| **Hologres** | Real-time interactive analytics on OSS data. |

## OSS as HDFS storage

Replace HDFS with OSS in Hadoop and Spark workloads using the `oss://` URI scheme:

```
spark.read.parquet("oss://data-lake-bucket/events/year=2025/")
```

Benefits over HDFS:
- No cluster storage management or rebalancing.
- Independent scaling of compute and storage.
- Built-in 11 nines of durability.
- Lower cost for infrequently accessed data with storage class tiering.

## Data formats

| Format | Best For |
|--------|---------|
| Parquet | Columnar analytics with predicate pushdown |
| ORC | Hive-optimized columnar format |
| CSV/JSON | Simple data exchange and ingestion |
| Avro | Schema evolution and row-oriented processing |

<Tip>
Use columnar formats (Parquet or ORC) with partitioned directory layouts for the best query performance and lowest data scan costs.
</Tip>

## Best practices

- **Partition data** by date, region, or category using directory prefixes (e.g., `year=2025/month=03/`).
- **Use Infrequent Access** storage class for data that is queried less than once per month.
- **Enable versioning** to protect against accidental overwrites in shared data lake environments.
- **Set lifecycle rules** to archive or expire old partitions automatically.
