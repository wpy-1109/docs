## Migration tools

| Tool | Use Case | Scale | Method |
|------|----------|-------|--------|
| **ossutil** | CLI-based batch operations | Small to medium | Online |
| **OSS Migration Service** | Cross-cloud or on-premises migration | Large | Online |
| **Data Transport (Lightning Cube)** | Physical device transfer | Terabytes to petabytes | Offline |
| **ossimport** | Distributed migration tool | Large | Online |
| **Cross-Region Replication** | OSS-to-OSS across regions | Continuous | Online |
| **Rclone** | Third-party S3-compatible tool | Various | Online |
| **Back-to-origin** | Transparent proxy during migration | N/A | Online |

## Migration planning

1. **Assess data volume**: Calculate total size, object count, and largest object
2. **Estimate time**: Factor in bandwidth, concurrency limits, and retry overhead
3. **Plan incrementally**: Use incremental sync to minimize cutover time
4. **Validate integrity**: Compare checksums (CRC-64 or Content-MD5) after transfer
5. **Test applications**: Verify application compatibility before switching production traffic
6. **Plan rollback**: Keep source data accessible during the transition period

## Using back-to-Origin for zero-Downtime migration

Configure back-to-origin rules so that OSS transparently fetches objects from the source on cache miss:

1. Set up back-to-origin rules pointing to the source storage
2. Switch DNS or application endpoints to OSS
3. On first access, OSS fetches from the source and caches locally
4. Run batch migration in the background to proactively populate OSS
5. Remove back-to-origin rules once migration is complete
