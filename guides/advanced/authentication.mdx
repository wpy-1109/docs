Every request to OSS must be authenticated (unless accessing public-read resources anonymously). OSS supports multiple authentication methods to verify the identity of the requester.

## Signature versions

| Version | Description | Recommendation |
|---------|-------------|----------------|
| **Signature V4** | Region-aware, time-limited signatures with improved security | Recommended for all new applications |
| **Signature V1** | Legacy signature format | Supported but not recommended |

<Info>
All current SDKs default to Signature V4. If you are using an older SDK version, upgrade to enable V4 support.
</Info>

## Credential types

| Credential | Use Case |
|-----------|----------|
| AccessKey pair (AK/SK) | Long-term credentials for server-side applications |
| STS temporary credentials | Short-lived tokens for untrusted clients or cross-account access |
| Instance RAM role | Automatic credential management for ECS, Function Compute, and other Alibaba Cloud services |
| OIDC-based SSO | Federated identity for enterprise SSO integration |

## Environment variable credentials

All OSS SDKs support loading credentials from environment variables:

```bash
export OSS_ACCESS_KEY_ID=<your-access-key-id>
export OSS_ACCESS_KEY_SECRET=<your-access-key-secret>
```

<Warning>
Never embed AccessKey credentials directly in source code. Use environment variables, instance RAM roles, or a credentials file.
</Warning>

## Signed URLs

Generate presigned URLs that grant temporary access to private objects without requiring the requester to have OSS credentials:

```
https://bucket.oss-cn-hangzhou.aliyuncs.com/file.pdf?x-oss-signature-version=OSS4-HMAC-SHA256&x-oss-credential=...&x-oss-expires=3600&x-oss-signature=...
```

## Best practices

- Use **Signature V4** for all new applications.
- Use **STS temporary credentials** for mobile apps, web browsers, and third-party integrations.
- Use **instance RAM roles** for applications running on Alibaba Cloud services.
- Rotate AccessKey pairs regularly and delete unused keys.
