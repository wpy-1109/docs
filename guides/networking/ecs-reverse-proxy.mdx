OSS endpoints use dynamic IP addresses, which can complicate firewall whitelisting and integrations that require a fixed IP. Deploy an Nginx reverse proxy on an ECS instance with a static public IP to provide a stable access address for your OSS resources.

## How it works

1. A client sends a request to the ECS instance's static public IP.
2. Nginx receives the request and forwards it to the target OSS bucket.
3. OSS returns the response to Nginx, which relays it to the client.

<Tip>
Place the ECS instance in the same region as your bucket to use the internal endpoint. This reduces latency and avoids public internet traffic costs.
</Tip>

## Prerequisites

- An ECS instance with a static public IP or Elastic IP Address (EIP).
- Nginx installed on the ECS instance.
- A security group rule allowing inbound HTTP (port 80) or HTTPS (port 443) traffic.

## Configure Nginx

Add a server block to your Nginx configuration that proxies requests to your OSS bucket:

```nginx
server {
    listen 80;
    server_name your-domain.com;

    location / {
        proxy_pass http://examplebucket.oss-cn-hangzhou-internal.aliyuncs.com;
        proxy_set_header Host examplebucket.oss-cn-hangzhou-internal.aliyuncs.com;
    }
}
```

After updating the configuration, reload Nginx:

```bash
sudo nginx -s reload
```

<Warning>
The `proxy_set_header Host` directive must match the OSS bucket domain. If the Host header does not match, OSS cannot identify the target bucket and returns an error.
</Warning>

## Best practices

- **Use the internal endpoint** when the ECS instance and bucket are in the same region to avoid public internet traffic fees.
- **Enable HTTPS** by configuring an SSL certificate on Nginx for encrypted client connections.
- **Set up health checks** if deploying behind a load balancer for high availability.
- **Configure access logging** in Nginx to audit and troubleshoot requests.
