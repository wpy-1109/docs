Buckets and objects in OSS must follow specific naming rules. Understanding these conventions helps you avoid errors and design a scalable object key structure.

## Bucket naming rules

| Rule | Requirement |
|------|-------------|
| **Length** | 3 to 63 characters |
| **Allowed characters** | Lowercase letters (`a-z`), digits (`0-9`), and hyphens (`-`) |
| **Start/end character** | Must start and end with a lowercase letter or digit |
| **Uniqueness** | Must be globally unique across all of OSS |
| **Immutable** | Cannot be renamed after creation |

### Valid bucket names

```
my-app-data
project2024-logs
cdn-assets-prod
user-uploads-cn-hangzhou
```

### Invalid bucket names

| Name | Reason |
|------|--------|
| `My-Bucket` | Contains uppercase letters |
| `-my-bucket` | Starts with a hyphen |
| `my-bucket-` | Ends with a hyphen |
| `my..bucket` | Contains consecutive dots |
| `ab` | Fewer than 3 characters |

<Tip>
Include your project name, environment, and region in bucket names to keep them organized. For example: `myapp-prod-cn-hangzhou` or `analytics-staging-us-west`.
</Tip>

## Object key naming rules

| Rule | Requirement |
|------|-------------|
| **Encoding** | UTF-8 |
| **Length** | 1 to 1,023 bytes |
| **Start character** | Must not start with `/` or `\` |
| **Case sensitivity** | Keys are case-sensitive |

### Characters to avoid

While OSS supports most UTF-8 characters in object keys, avoid these characters to prevent issues with certain tools and APIs:

| Character | Issue |
|-----------|-------|
| `\` (backslash) | Cannot be used as the first character. May cause parsing issues. |
| Control characters (0x00-0x1F, 0x7F) | May be misinterpreted by HTTP clients. |
| `#` (hash) | Interpreted as URL fragment. |
| `?` (question mark) | Interpreted as query string delimiter. |

### Recommended key patterns

Use forward slashes (`/`) to create a logical hierarchy in your keys:

```
# By content type
images/profile/user-12345.jpg
documents/reports/2024/q1-summary.pdf

# By date
logs/2024/03/15/access.log
backups/db/2024-03-15/dump.sql.gz

# By user/tenant
tenants/acme-corp/uploads/invoice.pdf
users/u-789/avatars/profile.png
```

## Performance considerations

<Warning>
Avoid sequential key prefixes (such as timestamps or auto-incrementing IDs) for high-throughput workloads. Sequential prefixes concentrate writes on a single partition, which limits performance.
</Warning>

**Instead of**:
```
2024-03-15-00001.log
2024-03-15-00002.log
2024-03-15-00003.log
```

**Use random or hashed prefixes**:
```
a3f2/2024-03-15-00001.log
7b1c/2024-03-15-00002.log
e9d4/2024-03-15-00003.log
```

You can compute a short hash prefix from the key to distribute objects across partitions:

```python
import hashlib

def add_prefix(key):
    prefix = hashlib.md5(key.encode()).hexdigest()[:4]
    return f"{prefix}/{key}"
```
