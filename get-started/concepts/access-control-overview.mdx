OSS provides a multi-layered access control model that lets you secure your data at different levels. You can combine these mechanisms to implement least-privilege access for your users and applications.

## Access control mechanisms

<CardGroup cols={2}>
  <Card title="Block public access" icon="ban">
    A master switch that blocks all public access to your bucket, overriding any other permissions. Enabled by default.
  </Card>
  <Card title="Bucket policies" icon="file-shield">
    Resource-based policies that grant fine-grained access to specific users, accounts, or anonymous users.
  </Card>
  <Card title="RAM policies" icon="user-shield">
    User-based policies that control which OSS resources your RAM users and roles can access.
  </Card>
  <Card title="Bucket ACL" icon="list-check">
    Simplified preset permissions (private, public-read, public-read-write) applied at the bucket level.
  </Card>
</CardGroup>

## Enforcement order

When a request arrives, OSS evaluates access control in this order:

<Steps>
  <Step title="Block public access">
    If Block Public Access is enabled and the request is unauthenticated, the request is **denied**.
  </Step>
  <Step title="Bucket policy">
    OSS checks bucket policies. An explicit **Deny** stops evaluation. An explicit **Allow** grants access.
  </Step>
  <Step title="RAM policy">
    If no bucket policy matches, OSS checks RAM policies attached to the user or role.
  </Step>
  <Step title="Bucket ACL">
    If no RAM policy matches, OSS falls back to the bucket ACL setting.
  </Step>
</Steps>

<Warning>
An explicit **Deny** in any policy always takes precedence over any Allow. Design your policies with this in mind.
</Warning>

## Comparison

| Feature | Block Public Access | Bucket Policy | RAM Policy | Bucket ACL | STS |
|---------|-------------------|---------------|------------|------------|-----|
| **Scope** | Bucket | Bucket + objects | User/role | Bucket | Temporary session |
| **Granularity** | On/off switch | Fine-grained (resource, action, condition) | Fine-grained | 3 presets | Inherits from RAM policy |
| **Target** | All public access | Specific users, accounts, IP ranges | RAM users/roles | All requests | Temporary credentials |
| **Use case** | Prevent any public access | Cross-account sharing, IP restrictions | Internal user permissions | Simple public/private control | Mobile/web clients, short-lived access |

## Block Public Access

Block Public Access is a safety mechanism that prevents public access to your bucket, regardless of bucket policies, ACLs, or object ACLs. It is enabled by default for new buckets created in the console.

<Warning>
Before disabling Block Public Access, verify that your bucket policies and ACLs do not expose sensitive data to the public.
</Warning>

## RAM policies

RAM (Resource Access Management) policies are attached to users, groups, or roles under your Alibaba Cloud account. They define which OSS actions a user can perform on which resources.

**Example**: Allow a RAM user to read objects in a specific bucket:

```json
{
  "Version": "1",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "oss:GetObject",
        "oss:ListObjects"
      ],
      "Resource": [
        "acs:oss:*:*:my-bucket",
        "acs:oss:*:*:my-bucket/*"
      ]
    }
  ]
}
```

For more details, see [RAM Policies](/guides/access-control/ram-policies).

## Bucket policies

Bucket policies are attached to a specific bucket and can grant access to other Alibaba Cloud accounts or anonymous users. They support conditions such as IP address restrictions and HTTPS requirements.

**Example**: Allow read access from a specific IP range:

```json
{
  "Version": "1",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": ["oss:GetObject"],
      "Resource": ["acs:oss:*:*:my-bucket/*"],
      "Condition": {
        "IpAddress": {
          "acs:SourceIp": "192.168.1.0/24"
        }
      }
    }
  ]
}
```

For more details, see [Bucket Policies](/guides/access-control/bucket-policies).

## Bucket ACL

Bucket ACLs provide three simple presets:

| ACL | Read | Write | Use Case |
|-----|------|-------|----------|
| **Private** | Owner only | Owner only | Default. Most secure option. |
| **Public-read** | Anyone | Owner only | Publicly readable content (e.g., static website assets). |
| **Public-read-write** | Anyone | Anyone | Not recommended. Use only for public upload scenarios with caution. |

<Warning>
Avoid `public-read-write` in production. Use presigned URLs or STS temporary credentials for controlled upload access instead.
</Warning>

## STS temporary credentials

Security Token Service (STS) provides temporary credentials with a limited lifespan and scoped permissions. Use STS when you need to grant short-lived access to untrusted clients, such as mobile apps or web browsers.

Benefits:
- Credentials expire automatically (15 minutes to 12 hours by default)
- Scoped to specific actions and resources
- No need to expose your permanent AccessKey pair

For more details, see [STS Temporary Credentials](/guides/access-control/sts-temporary-credentials).

## Presigned URLs

Presigned URLs provide time-limited access to a specific object without requiring authentication. Generate a presigned URL on your server and share it with clients for secure download or upload.

For more details, see [Presigned URLs](/guides/access-control/presigned-urls).

## Best practices

<Tip>
- Keep **Block Public Access** enabled unless you explicitly need public access.
- Use **RAM policies** for internal users and applications -- follow the principle of least privilege.
- Use **STS temporary credentials** for mobile and web clients.
- Use **bucket policies** for cross-account access and IP-based restrictions.
- Avoid **public-read-write** ACLs. Use presigned URLs for controlled uploads.
- Enable **access logging** to audit who accesses your data.
</Tip>
