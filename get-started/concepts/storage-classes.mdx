OSS provides five storage classes to match different data access patterns and cost requirements. By choosing the right storage class, you can reduce storage costs by up to 80% while maintaining the durability and availability your application needs.

## Storage class overview

<CardGroup cols={2}>
  <Card title="Hot data" icon="fire">
    **Standard** -- For frequently accessed data that requires low-latency reads. No minimum storage duration or object size.
  </Card>
  <Card title="Warm data" icon="temperature-half">
    **Infrequent Access (IA)** -- For data accessed once or twice a month. Lower storage cost with per-GB retrieval fees. 30-day minimum.
  </Card>
  <Card title="Cold data" icon="snowflake">
    **Archive** -- For data accessed rarely (once or twice a year). Must be restored before access unless direct read is enabled. 60-day minimum.
  </Card>
  <Card title="Frozen data" icon="icicles">
    **Cold Archive / Deep Cold Archive** -- For data accessed almost never. Lowest storage cost, longest restore times (hours to days). 180-day minimum.
  </Card>
</CardGroup>

### Standard

Standard storage provides high-performance, low-latency access for frequently accessed data. It is the default storage class and is suitable for application data, websites, streaming media, mobile apps, and big data analytics. Standard supports both zone-redundant storage (ZRS) and locally redundant storage (LRS) with no minimum object size or storage duration requirements.

### Infrequent Access (IA)

Infrequent Access is designed for data accessed less than once a month but still requiring real-time reads when needed. IA has a lower storage price than Standard but charges per-GB retrieval fees. The minimum billable object size is 64 KB and the minimum storage duration is 30 days. IA supports both ZRS and LRS. Typical use cases include logs, monitoring data, and periodic backups.

### Archive

Archive is designed for data accessed less than once per quarter. Archive has a minimum billable object size of 64 KB and a minimum storage duration of 60 days. Archive supports both ZRS and LRS. By default, Archive objects must be restored before access. With [Archive Direct Read](/guides/data-management/archive-direct-read) enabled, you can read Archive objects in real time without restoring them first. Typical use cases include compliance records, medical imaging, and scientific data.

### Cold Archive

Cold Archive provides very low-cost storage for data that is rarely accessed and can tolerate restore times of 1 to 12 hours. Cold Archive has a minimum billable object size of 64 KB and a minimum storage duration of 180 days. Cold Archive supports LRS only. Typical use cases include tape replacement and long-term regulatory archives.

### Deep Cold Archive

Deep Cold Archive offers the lowest storage cost for data that is almost never accessed and can tolerate restore times of 12 to 48 hours. It has a minimum billable object size of 64 KB and a minimum storage duration of 180 days. Deep Cold Archive supports LRS only. Typical use cases include genomic data, historical records, and disaster recovery copies.

## Data redundancy options

OSS offers two data redundancy mechanisms. Choose based on your availability and durability requirements.

**Zone-redundant storage (ZRS)** distributes your data across multiple zones in the same region. Even if an entire zone becomes unavailable, your data remains accessible. ZRS provides 12 nines of data durability and up to 99.995% service availability.

**Locally redundant storage (LRS)** stores multiple copies of your data on different devices within a single zone. LRS provides 11 nines of data durability and up to 99.99% service availability. If the zone becomes unavailable, data in that zone is inaccessible until the zone recovers.

| Storage Class | ZRS Support | LRS Support |
|--------------|-------------|-------------|
| Standard | Yes | Yes |
| Infrequent Access | Yes | Yes |
| Archive | Yes | Yes |
| Cold Archive | No (LRS only) | Yes |
| Deep Cold Archive | No (LRS only) | Yes |

<Tip>
For production workloads that require high availability, choose ZRS. ZRS is available in China (Hangzhou, Shanghai, Beijing, Zhangjiakou, Ulanqab, Shenzhen), China (Hong Kong), Japan (Tokyo), Singapore, Indonesia (Jakarta), Germany (Frankfurt), and US (Virginia) for Standard, IA, and Archive storage classes.
</Tip>

## Choosing a storage class

Use this decision guide to select the right storage class based on your data access patterns.

```
How often is this data accessed?
│
├── Multiple times per day ──────────────► Standard
│
├── A few times per month ───────────────► Infrequent Access (IA)
│
├── A few times per year ────────────────► Archive
│
├── Rarely, can wait hours ──────────────► Cold Archive
│
└── Rarely, can wait 1-2 days ───────────► Deep Cold Archive
```

| Access Pattern | Storage Class | Restore Time | Best For |
|---|---|---|---|
| Multiple times per day | **Standard** | Instant | Active application data, websites |
| A few times per month | **Infrequent Access** | Instant | Logs, monitoring data, backups |
| A few times per year | **Archive** | ~1 minute (standard restore) | Compliance records, media archives |
| Rarely, can wait hours | **Cold Archive** | 1-12 hours | Tape replacement, long-term archives |
| Rarely, can wait longer | **Deep Cold Archive** | 12-48 hours | Genomics, historical records |

## Restore times and access patterns

Archive, Cold Archive, and Deep Cold Archive objects must be restored before you can read them. Each class offers multiple restore priorities that trade speed for cost.

| Storage Class | Restore Priority | Typical Restore Time | Relative Cost |
|--------------|-----------------|---------------------|---------------|
| **Archive** | Standard | ~1 minute | Low |
| **Cold Archive** | Expedited | Within 1 hour | Highest |
| **Cold Archive** | Standard | 2-5 hours | Medium |
| **Cold Archive** | Bulk | 5-12 hours | Lowest |
| **Deep Cold Archive** | Expedited | Within 12 hours | Highest |
| **Deep Cold Archive** | Standard | Within 48 hours | Lowest |

Higher restore priorities incur higher data retrieval fees. Use bulk restore for non-urgent access to minimize costs. See [Restore archived objects](/guides/objects/restore-objects) for detailed instructions.

## Minimum storage commitments

<Warning>
If you delete, overwrite, or transition an object before the minimum storage duration has elapsed, you are still charged for the remaining days of the minimum period. Factor this into your cost calculations when using lifecycle rules.
</Warning>

| Storage Class | Minimum Duration | Minimum Object Size | Early Deletion Charge |
|--------------|-----------------|--------------------|-----------------------|
| Standard | None | None | None |
| Infrequent Access | 30 days | 64 KB | Charged for remaining days |
| Archive | 60 days | 64 KB | Charged for remaining days |
| Cold Archive | 180 days | 64 KB | Charged for remaining days |
| Deep Cold Archive | 180 days | 64 KB | Charged for remaining days |

Objects smaller than 64 KB stored in IA, Archive, Cold Archive, or Deep Cold Archive are billed as 64 KB objects.

## Automatic tiering with lifecycle rules

You can use lifecycle rules to automatically transition objects between storage classes based on age. This lets you store data in the most cost-effective tier without manual intervention.

```
Standard  →  (after 30 days)  →  IA  →  (after 90 days)  →  Archive  →  (after 365 days)  →  Cold Archive
```

Configure lifecycle rules to match your data access patterns. For example, log files that are actively analyzed for the first month, referenced occasionally for three months, and then retained for compliance can follow the transition pattern above.

<Note>
You can only transition objects from a higher-access tier to a lower-access tier (Standard to IA to Archive to Cold Archive to Deep Cold Archive). To move objects to a higher tier, you must copy or overwrite them with the desired storage class.
</Note>

For configuration details, see [Lifecycle rules](/guides/data-management/lifecycle-rules).
