# OSS AccessKey 安全管理与泄露应急处理

发现 OSS 流量异常暴增或收到异常扣费通知，怀疑 AccessKey 被泄露。本文介绍 AK 安全管理的最佳实践和泄露后的应急处理步骤。

## 适用场景

- OSS 外网流出流量突然暴增，产生异常费用。
- 代码仓库（如 GitHub）中意外提交了 AK/SK。
- 日志中发现非预期的 AccessKey 在访问 Bucket。
- 收到阿里云安全告警，提示 AK 疑似泄露。

## AK 泄露应急处理

### 1. 立即禁用或删除泄露的 AccessKey

1. 登录 RAM 控制台 > 身份管理 > 用户。
2. 找到泄露 AK 对应的 RAM 用户。
3. 在 AccessKey 页签中，**禁用**该 AK（如果暂时不确定影响，先禁用而非删除）。
4. 确认业务不受影响后，**删除**该 AK。

### 2. 创建新的 AccessKey

1. 为对应 RAM 用户创建新的 AK。
2. 更新所有使用该 AK 的应用配置。
3. 重启相关服务。

### 3. 排查损失和影响

1. 进入 OSS 控制台 > 目标 Bucket > **日志管理** > **实时查询**。
2. 使用 `access_id` 字段筛选泄露的 AK 的操作记录。
3. 检查是否有异常的 `GetObject`（下载）、`PutObject`（上传）或 `DeleteObject`（删除）操作。
4. 查看费用中心的账单明细，评估异常费用。

### 4. 加固安全配置

- 将受影响的 Bucket 设为私有权限。
- 开启阻止公共访问。
- 配置 Bucket Policy 限制访问来源 IP/VPC。
- 开启防盗链。

## AccessKey 安全管理最佳实践

### 不要使用主账号 AK

主账号 AK 拥有所有资源的完全控制权限。一旦泄露，攻击者可以操作账号下的所有云资源。

**推荐做法**：
- 创建 RAM 用户，授予最小必要权限。
- 使用 RAM 用户的 AK 进行 API 调用。
- 主账号 AK 建议不创建或创建后立即禁用。

### 不要把 AK 硬编码在代码中

- 不要将 AK/SK 提交到代码仓库。
- 使用环境变量、配置文件或密钥管理服务（KMS）存储密钥。
- 在 `.gitignore` 中排除包含密钥的配置文件。

### 使用 STS 临时凭证

给客户端（App、小程序、Web 前端）提供 STS 临时凭证而非长期 AK：
- 临时凭证有有效期，过期自动失效。
- 可通过 Policy 参数限制临时凭证只能操作特定路径。

### 定期轮换 AccessKey

1. 创建新 AK。
2. 更新应用配置使用新 AK。
3. 确认旧 AK 不再使用后，禁用并删除。

### 开启操作审计

通过 ActionTrail 记录所有 API 操作，便于安全审计和异常排查。

## 常见问题

**Q: 日志中看到 `sts.` 开头的 AK 在访问我的 Bucket，这是什么？**

`STS.` 开头的 AccessKey 是 STS 临时凭证。这通常是你自己的业务通过 STS 方式访问 OSS。检查对应的 RAM 角色名称即可确认。

**Q: 发现一个不认识的 AK 在访问我的 Bucket，但这个 AK 不是我的？**

可能是 OSS 系统内部角色（如生命周期管理、数据复制等内部服务）使用的 AK。检查 Bucket 是否配置了生命周期规则或数据复制规则。

**Q: 流量被盗刷，费用能退吗？**

因 AK 泄露导致的流量盗刷产生的费用，阿里云通常无法退款（属于用户侧安全管理责任）。建议第一时间禁用泄露的 AK 止损，并加固安全配置防止再次发生。
