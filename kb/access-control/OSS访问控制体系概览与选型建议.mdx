# OSS 访问控制体系概览与选型建议

OSS 提供多种访问控制机制（ACL、Bucket Policy、RAM Policy、STS 等），不同场景应选择不同的方式。本文梳理各机制的定位和适用场景，帮助你选择合适的方案。

## 详细说明

### OSS 访问控制机制一览

| 机制 | 管理位置 | 管理维度 | 适用场景 |
|------|---------|---------|---------|
| **Bucket ACL** | OSS 控制台 | Bucket 级别 | 快速设置 Bucket 的公共/私有属性 |
| **Object ACL** | OSS 控制台/API | Object 级别 | 单独控制某个文件的访问权限 |
| **Bucket Policy** | OSS 控制台 | Bucket 级别，支持条件 | 跨账号授权、IP/VPC 限制、目录级控制 |
| **RAM Policy** | RAM 控制台 | 用户/角色级别 | 管控 RAM 用户和角色的 OSS 操作权限 |
| **STS 临时凭证** | 代码调用 | 临时、可限定范围 | 客户端直传、临时授权第三方 |
| **预签名 URL** | 代码生成 | Object 级别，有有效期 | 临时分享文件、外部下载链接 |
| **防盗链（Referer）** | OSS 控制台 | Bucket 级别 | 防止资源被第三方网站盗用 |
| **阻止公共访问** | OSS 控制台 | 账号/Bucket 级别 | 全局禁止公共读写 |

### 权限判断优先级

当多种机制同时生效时，OSS 按以下顺序判断：

1. **阻止公共访问** → 如果开启，所有公共 ACL 和匿名访问策略失效
2. **显式 Deny**（Bucket Policy 或 RAM Policy）→ 直接拒绝
3. **显式 Allow**（Bucket Policy 或 RAM Policy）→ 允许访问
4. **Bucket/Object ACL** → 判断是否允许匿名访问
5. **默认拒绝** → 以上都未匹配时，拒绝访问

### 场景选型建议

**场景 1：网站图片/视频托管**
- 推荐：CDN + 私有 Bucket 回源
- 理由：安全、快速、费用低

**场景 2：多团队共用一个账号，各团队访问独立 Bucket**
- 推荐：RAM 用户 + RAM 自定义权限策略
- 理由：集中管理、最小权限原则

**场景 3：给外部合作方临时授权访问**
- 推荐：STS 临时凭证 或 预签名 URL
- 理由：有时效性、不暴露长期密钥

**场景 4：跨账号数据共享**
- 推荐：Bucket Policy（授权其他账号 UID）
- 理由：无需在对方账号做额外配置

**场景 5：限制只有办公网络能访问**
- 推荐：Bucket Policy（IP + VPC 条件）
- 理由：网络层限制，简单有效

**场景 6：移动 App / 小程序直传**
- 推荐：STS 临时凭证
- 理由：安全、灵活、可精细控制目录范围

**场景 7：对外提供文件下载链接**
- 推荐：预签名 URL
- 理由：简单、安全、有有效期

## 常见问题

**Q: 我应该用 Bucket Policy 还是 RAM Policy？**

- 如果是管理"这个 Bucket 谁能访问"，用 Bucket Policy。
- 如果是管理"这个用户能访问哪些资源"，用 RAM Policy。
- 两者可以组合使用。

**Q: 我只是想让上传的图片能公开访问，最简单的方式是什么？**

最简单但不推荐的方式是将 Bucket ACL 设为公共读。推荐方式是保持私有 + 生成签名 URL 或使用 CDN 回源。

**Q: 授权做对了但还是报 AccessDenied？**

按以下顺序排查：
1. 检查"阻止公共访问"是否开启。
2. 检查是否有 Bucket Policy 的显式 Deny。
3. 检查 RAM Policy 是否有显式 Deny。
4. 确认 AK 和 Bucket 在同一账号。
5. 确认 Resource 字段格式正确（同时包含 bucket 和 bucket/*）。
