### Q1：AccessKey Secret 在哪里查看？

AccessKey Secret 只在创建时显示一次，之后无法再次查看。

**查找路径**：RAM 控制台 → 身份管理 → 用户 → 选择用户 → 认证管理 → AccessKey

如果忘记了 AccessKey Secret：
1. 在 RAM 控制台中为该用户创建新的 AccessKey
2. 使用新的 AK/SK 替换应用中的旧配置
3. 确认所有应用已切换到新 AK 后，禁用并删除旧 AccessKey

---

### Q2：Bucket 怎么设置 AccessKey？

Bucket 本身不设置 AccessKey。AccessKey 是账号级别的身份凭证，与 Bucket 无关。

配置方法：
1. 在 RAM 控制台创建用户并创建 AccessKey
2. 为该用户授予 OSS 权限（如 AliyunOSSFullAccess 或自定义策略）
3. 在 SDK 或工具中使用该 AccessKey 连接 OSS

第三方系统要求填写的 Access Key、Secret Key、Region 等参数对应关系：
| 第三方术语 | 阿里云 OSS 对应项 |
|-----------|------------------|
| Access Key / SecretId | AccessKey ID |
| Secret Key / SecretKey | AccessKey Secret |
| Region | 地域 ID（如 cn-hangzhou） |
| Endpoint | oss-cn-hangzhou.aliyuncs.com |
| Bucket Name / 存储桶名称 | Bucket 名称 |

### Q3：AccessKey 登录 OSS 显示"用户禁用"

错误信息：
```
错误代码: 用户禁用
RequestId: 69A02755F5E96E3739EC8D1E
ErrorCode: 0003-00000801L
```

**原因**：该 AccessKey 所属的 RAM 用户被禁用，或 AccessKey 本身被禁用。

**解决方案**：
1. 登录主账号的 RAM 控制台，检查该 RAM 用户的状态
2. 如果用户被禁用，启用该用户
3. 如果是 AccessKey 被禁用，启用该 AccessKey 或创建新的

### Q4：AK 泄露后如何排查访问记录？

当 AccessKey 疑似泄露时：

**紧急处理**：
1. 立即在 RAM 控制台**禁用**该 AccessKey（不要直接删除，先禁用以保留审计追溯能力）
2. 创建新的 AccessKey 替换所有使用旧 AK 的应用

**排查方法**：
1. **操作审计（ActionTrail）**：查看该 AK 的所有 API 调用记录，包括调用时间和来源 IP
2. **OSS 日志（Logging）**：如果 Bucket 开启了日志记录，可以查看该 AK 对 OSS 的详细操作记录
3. **注意**：OSS 部分操作事件在 ActionTrail 中可能显示为"未支持的事件"，这些通常是数据面操作（如 GetObject），需要通过 OSS 访问日志查看

### Q5：AccessKey 级网络访问限制策略对 OSS 是否生效？

**生效**。在 RAM 控制台中为 AccessKey 配置的网络访问限制策略（如限制 IP 白名单）对 OSS 的访问同样生效。配置后，只有从白名单 IP 发起的请求才能使用该 AccessKey 访问 OSS。

**注意事项**：
- 配置网络限制后，确保所有使用该 AK 的服务器 IP 都在白名单中
- 如果使用 NAT 网关或代理，需要添加出口 IP
- 配置变更后需要几分钟生效

---

### Q6：如何解除 AccessKey 的风控限制？

如果 AccessKey 被安全系统风控限制（如 OpenCLAW 限制），通常是因为检测到异常使用行为。

**解决方案**：
1. 登录 RAM 控制台查看该 AK 的安全事件
2. 确认是否存在 AK 泄露风险
3. 如果确认安全，可以提交工单申请解除风控限制
4. 如果有泄露风险，建议禁用该 AK 并创建新的

---

### Q7：如何查看 AccessKey 的使用情况？

1. **操作审计（ActionTrail）**：记录所有 API 调用，可按 AccessKey ID 过滤
2. **RAM 控制台**：显示 AccessKey 的最后使用时间和使用的云服务
3. **OSS 访问日志**：Bucket 开启日志后，可查看每次请求使用的 AccessKey ID

**限制**：ActionTrail 中 OSS 的部分数据面操作（如 GetObject/PutObject）可能显示为"未支持的事件"，且不包含来源 IP。完整的访问记录需要通过 OSS 的实时日志或日志转存功能获取。

---

### Q8：创建新 AccessKey 后，另一个账号的 OSS 操作失败了？

**原因**：这是典型的"AK 混用"问题。如果您在同一台机器上配置了多个阿里云账号的 AccessKey（如 ossutil、SDK 配置文件），新创建的 AK 可能覆盖了之前的配置。

**排查方法**：
1. 检查 ossutil 配置文件（默认 `~/.ossutilconfig`），确认 AK/SK 对应的是正确的账号
2. 检查环境变量 `OSS_ACCESS_KEY_ID` 和 `OSS_ACCESS_KEY_SECRET`
3. 确认每个 AK 只能访问其所属账号下的 Bucket

---

### Q9：跨账号复制 OSS 文件时认证失败

使用 ossutil 跨账号复制文件时，需要注意：

1. 源 Bucket 和目标 Bucket 的 AK/SK 不同，ossutil 不支持在单条命令中使用两套凭证
2. **推荐方案**：
   - 方案一：在源 Bucket 上设置 Bucket Policy，授权目标账号的 RAM 用户读取权限
   - 方案二：使用 OSS 跨区域复制（CRR）功能
   - 方案三：使用 Data Transport（在线迁移服务）

---

### Q10：使用 OSS Browser 连接时提示"看不到内容"

**常见原因**：
1. AccessKey 所属的 RAM 用户没有 ListBucket 权限
2. AccessKey 被禁用
3. Bucket 策略限制了访问来源

**解决方案**：
1. 在 OSS Browser 的设置中确认 Endpoint 和 AK/SK 正确
2. 为 RAM 用户添加 `oss:ListBucket` 和 `oss:ListObjects` 权限
3. 尝试使用 ossutil 命令行验证：`ossutil ls oss://bucket-name/`
