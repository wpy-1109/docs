# OSS 跨域（CORS）配置后仍然报错

## 症状

在 OSS 控制台已配置 CORS 跨域规则（允许来源设为 `*`），但浏览器访问 OSS 资源时仍报以下错误：

```
Access to fetch at 'https://bucket-name.oss-cn-xxx.aliyuncs.com/xxx.png'
from origin 'https://yourdomain.com' has been blocked by CORS policy:
No 'Access-Control-Allow-Origin' header is present on the requested resource.
```

或在浏览器 Console 中看到类似跨域拒绝的报错。

## 原因

常见原因有三类：

1. **未勾选"返回 Vary: Origin"**：这是最常见的原因。未勾选时，浏览器或 CDN 可能缓存了一个不带 CORS 响应头的版本，后续跨域请求复用缓存后即报错。
2. **浏览器缓存了旧响应**：修改 CORS 规则后，浏览器仍使用之前缓存的无 CORS 头响应。
3. **暴露 Header 未正确设置**：部分 SDK 或前端框架需要读取 `ETag`、`x-oss-request-id` 等响应头，若未在暴露 Header 中列出，浏览器会拦截。
4. **CDN 缓存未刷新**：如果使用了 CDN 加速 OSS，CDN 节点可能缓存了不带 CORS 头的旧响应。
5. **请求实际命中了防盗链规则**：返回 403 时不会带 CORS 头，浏览器将其报告为跨域错误。

## 解决方案

### 方案 1：标准 CORS 配置（推荐）

登录 OSS 控制台，在 Bucket 的「数据安全」>「跨域设置」中按以下参数配置：

| 配置项 | 值 |
|--------|-----|
| 来源 | `*`（或指定域名如 `https://yourdomain.com`） |
| 允许 Methods | 全部勾选（GET、POST、PUT、DELETE、HEAD） |
| 允许 Headers | `*` |
| 暴露 Headers | `ETag`（换行）`x-oss-request-id` |
| 返回 Vary: Origin | **必须勾选** |
| 缓存时间 | 0（调试期间设为 0，正式可设为 600） |

> **关键点**：暴露 Headers 中的 `ETag` 和 `x-oss-request-id` 需要分行填写，不要写在同一行。

### 方案 2：清除浏览器缓存

配置完 CORS 规则后，必须执行以下操作之一：
- 清除浏览器缓存后重新访问
- 使用无痕/隐私模式测试
- 换一个浏览器验证

### 方案 3：CDN 场景额外处理

如果 OSS 前面有 CDN，还需要：
1. 在 CDN 控制台刷新对应 URL 或目录的缓存
2. 在 CDN 的「HTTP 配置」>「自定义响应头」中添加 `Access-Control-Allow-Origin`

### 方案 4：排查防盗链冲突

检查 Bucket 是否配置了防盗链（Referer 白名单）：
- 如果启用了防盗链且不允许空 Referer，浏览器的预检（OPTIONS）请求不带 Referer，会被拒绝
- 将来源域名加入防盗链白名单，或允许空 Referer

## 预防建议

- 建议始终勾选「返回 Vary: Origin」，这样浏览器会按 Origin 值分别缓存响应
- 若有多个来源域名，可在 CORS 规则中分别添加，而非使用 `*`
- 调试时先用 `curl -H "Origin: https://yourdomain.com" -v` 测试 OSS 是否正确返回 CORS 头
