# CDN/ESA 加速 OSS 的架构配置与流量异常排查

## Q1：CDN 和 OSS 传输加速有什么区别？该选哪个？

| 对比项 | CDN 加速 | OSS 传输加速 |
|--------|---------|-------------|
| **原理** | 将资源缓存到全球边缘节点 | 优化 OSS 的传输链路（骨干网加速） |
| **适用场景** | 静态资源分发（图片、视频、JS/CSS） | 上传/下载大文件到 OSS |
| **缓存** | 有缓存，减少回源 | 无缓存，每次直连 OSS |
| **费用** | CDN 流量费（通常比 OSS 外网流量便宜） | OSS 传输加速流量费 |
| **推荐** | 读多写少的场景 | 写入或大文件传输场景 |

> **注意**：OSS 加速器（Data Accelerator）是另一个产品，用于加速内网大数据读取，与 CDN 和传输加速都不同。加速器只能走内网访问，不适用于公网场景。

## Q2：接入了 CDN/ESA 后，OSS 仍然产生外网流量费？

**排查步骤**：
1. **检查是否仍有直接访问 OSS 域名的请求**：
   - 查看 OSS 实时日志，关注 `sync_request` 字段
   - `sync_request: cdn` 表示 CDN/ESA 回源请求
   - 字段为空表示直接访问 OSS 域名

2. **检查所有访问 URL 是否已切换到 CDN 域名**：应用中可能残留了 OSS 默认域名的 URL

3. **检查搜索引擎/爬虫是否直接爬取 OSS URL**：将 Bucket 设为私有权限可阻止

4. **CDN 回源也会产生 OSS 流量**：缓存未命中时 CDN 需要回源拉取，这部分是正常的

## Q3：OSS 能自动刷新 CDN/ESA 缓存吗？

- **CDN**：OSS 更新文件后不会自动刷新 CDN 缓存，需要手动提交刷新请求或通过 API 刷新
- **ESA**：目前 OSS 不支持自动刷新 ESA 缓存（功能尚未支持）

**建议方案**：
- 文件更新时通过 CDN API 提交 URL 刷新或目录刷新
- 使用文件版本化 URL（如 `image.jpg?v=202603`）避免缓存问题
- 使用函数计算（FC）监听 OSS 事件通知，自动触发 CDN 刷新

## Q4：多个 CDN 域名指向同一个 OSS，更新文件后需要都刷新吗？

是的。CDN 缓存按加速域名隔离，每个域名有独立的缓存。文件更新后，需要对每个 CDN 域名分别提交刷新请求。目前不支持一次刷新让所有关联域名同时生效。

## Q5：一个 OSS Bucket 可以挂多个 CDN/ESA 域名吗？

可以。一个 Bucket 可以被多个 CDN 或 ESA 加速域名作为源站。每个加速域名独立管理缓存和配置。

## Q6：ESA 和 CDN 有什么区别？

| 对比项 | CDN | ESA（边缘安全加速） |
|--------|-----|-------------------|
| **功能** | 内容分发加速 | 加速 + 安全防护（WAF、DDoS、Bot 管理） |
| **回源方式** | 支持公共/私有 Bucket | 支持公共/私有 Bucket |
| **缓存刷新** | 支持 API 刷新 | OSS 暂不支持自动刷新 |
| **适用场景** | 纯加速需求 | 加速 + 安全防护需求 |

## Q7：OSS 流量突然暴增怎么排查？

**排查步骤**：
1. 在 OSS 控制台查看「监控服务」的带宽/流量图表，定位异常时间段
2. 使用「日志分析」或实时日志查询，找出：
   - 流量最大的文件（Top Object）
   - 请求次数最高的来源 IP
   - 非 200 请求的比例（非 200 请求不计流量费）
3. 判断是否被盗刷：
   - 单个 IP 请求次数异常高 → 疑似爬虫/恶意访问
   - 大量不同 IP 同时请求同一文件 → 可能是热点资源被引用

**应急措施**：
- 配置 Referer 防盗链限制访问来源
- 使用 Bucket Policy 限制特定 IP 或 IP 段
- 接入 CDN/ESA 后配置频次控制和 WAF 规则
- 参考「OSS 网站资源被盗刷导致流量费用异常的应急处理」

## Q8：CDN 域名和 OSS 自定义域名冲突吗？

不冲突，但同一个域名只能选择一种接入方式：
- 如果域名 CNAME 到 CDN，则通过 CDN 访问 OSS
- 如果域名 CNAME 到 OSS，则直接访问 OSS

**不要同时在 OSS 和 CDN 都绑定同一个域名的 CNAME**，这会导致配置冲突。ESA 的域名绑定也是同理。
