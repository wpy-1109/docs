### 适用场景

- 需要定期将本地文件增量同步到 OSS
- 大量小文件的定时备份场景
- 需要选择 `cp` 和 `sync` 命令进行数据同步

### 前提条件

- 已安装 ossutil 命令行工具
- 已配置 AccessKey 和 Endpoint

### cp 与 sync 命令对比

| 对比项 | `cp` 命令 | `sync` 命令 |
|--------|-----------|-------------|
| 工作方式 | 边扫描源端，边拷贝 | 同时扫描源和目的端，然后再拷贝 |
| `--delete` 选项 | 不支持 | 支持（删除目的端多余文件） |
| 源文件数量限制 | 无限制 | 默认 200 万，最大 500 万 |
| 增量模式 | 加 `-u` 参数 | 加 `-u` 参数 |
| 适用场景 | 大量文件增量上传 | 双向同步、镜像备份 |

### 增量同步操作步骤

#### 使用 cp 命令增量同步（推荐）

```bash
ossutil cp -r /local/data/ oss://your-bucket/backup/ -u -f
```

关键参数说明：
- `-r`：递归上传目录
- `-u`：增量模式，仅上传目标端不存在或本地文件修改时间更新的文件
- `-f`：强制执行，不提示确认

#### 增量判断逻辑

使用 `-u` 参数时，ossutil 通过比对源文件和 OSS 中同名文件的**最后修改时间**来判断：
- 本地文件修改时间 **晚于** OSS 中同名文件 → 重新上传
- 本地文件修改时间 **早于或等于** OSS 中同名文件 → 跳过

#### 硬链接场景的注意事项

OSS 是扁平化存储，不支持硬链接概念。如果本地使用硬链接做增量备份（如每日备份目录下的硬链接文件），上传到 OSS 时**每个路径下的文件都会被当作独立文件上传**，即使它们在本地指向同一个 inode。

**解决方案：**
- 固定上传目录路径，使用 `-u` 参数避免重复上传未修改的文件
- 不要将每日备份的不同目录都上传到 OSS（否则存储量会成倍增长）

### 验证

上传完成后，使用以下命令验证：

```bash
# 列出已上传的文件
ossutil ls oss://your-bucket/backup/ --limited-num 10
```

### 常见问题

**Q：ossimport 和 ossutil 该用哪个？**
ossimport 工具已不再维护，建议使用 ossutil。对于大规模数据迁移场景，也可以考虑在线迁移服务。
