### Q1：上传同名文件会怎样？

**回答：** OSS 默认执行**覆盖操作**。新上传的文件会直接替换同名旧文件，且该操作不可逆。无论是通过控制台、SDK 还是 API 上传，行为一致。

如需防止覆盖：
- 开启**版本控制**，旧版本会被保留
- 在业务层生成唯一文件名（如添加 UUID 或时间戳）
- 使用 `x-oss-forbid-overwrite` 请求头禁止覆盖

### Q2：用 ossutil 上传文件夹后，目录结构丢失（文件散落）怎么办？

**回答：** 这通常是 ossutil cp 命令中目标路径的写法问题。

**错误示例：** 上传 `AO/` 目录到 `oss://bucket/MKD/`，但目标路径未包含源目录名称，导致 `AO/` 下的文件直接散落在 `MKD/` 下。

```bash
# 错误：AO 下的文件会直接放到 MKD/ 下，没有 AO 目录
ossutil cp -r E:\跑图\可\AO\ oss://bucket/MKD/ -u -f

# 正确：在目标路径中加上源目录名称
ossutil cp -r E:\跑图\可\AO\ oss://bucket/MKD/AO/ -u -f
```

### Q3：为什么 ossutil 上传后生成了多余的文件夹？

**回答：** OSS 是扁平化存储，没有真正的目录概念。控制台中显示的"文件夹"实际上是根据 Key 中的 `/` 分隔符模拟出来的。ossutil 上传时如果源路径和目标路径写法不当，可能导致出现非预期的目录层级。

建议在上传前先用 `--dry-run` 参数模拟执行，确认文件路径符合预期。

### Q4：上传后文件后缀变成大写（如 .pdf 变成 .PDF），是怎么回事？

**回答：** OSS 不会修改文件名。上传后的文件名与上传时指定的 Object Key 完全一致。如果发现后缀大小写变化，请检查上传时代码中定义的文件名是否存在大小写转换逻辑。

### Q5：OSS 上传文件显示大小与本地不一致（如 25GB 显示为 23GB）

**回答：** 这是存储单位换算差异。操作系统可能以 1000 为基数（GB），而 OSS 使用 1024 为基数（GiB）。

```
25,585,017,551 字节 ÷ 1024 ÷ 1024 ÷ 1024 ≈ 23.83 GiB
```

文件数据不会丢失，大小是一致的。

### Q6：使用 SDK 上传文件时，OSS 不返回存储路径？

**回答：** 这是正常行为。使用阿里云 OSS SDK 上传文件时，成功后返回的是 HTTP 200 状态码和 RequestId，不会返回文件的完整存储路径。

文件的存储路径就是您上传时指定的 Object Key（即 Bucket 名称 + Object 名称组成的路径）。您需要在客户端自行拼接和记录该路径。
