## V2 SDK benefits

V2 SDKs provide significant improvements over V1:

- **Simplified authentication** -- built-in credential providers with automatic rotation
- **Automatic retry** -- configurable retry with exponential backoff
- **Paginator** -- iterate over paginated results without manual pagination
- **Transmission managers** -- high-level upload/download with automatic multipart and parallelism
- **File-like operations** -- read and write objects as file-like streams
- **Improved error handling** -- structured error types with detailed context

<Note>
V1 SDKs remain supported but are in maintenance mode. New features are only added to V2 SDKs. Migrate to V2 when possible.
</Note>

## SDK version lifecycle

| Phase | Description | Duration |
|-------|-------------|----------|
| **Active** | Full feature development, bug fixes, and security patches | Ongoing for latest major version |
| **Maintenance** | Security patches and critical bug fixes only; no new features | Typically 12-24 months after successor release |
| **End of Life** | No updates; upgrade required | After maintenance period |

## Authentication requirements

All SDKs must use one of the following authentication methods:

### Recommended

1. **STS temporary credentials** -- for client-side and short-lived access
2. **Instance RAM roles** -- for applications running on Alibaba Cloud ECS/ECI
3. **Managed Credentials Plugin** -- for server-side applications with automatic key rotation

### Acceptable

4. **RAM user AccessKey** -- for server-side applications in controlled environments

### Not permitted

5. **Root account AccessKey** -- never use the primary Alibaba Cloud account credentials
6. **Hardcoded credentials** -- never embed AccessKeys in source code

<Warning>
Using root account credentials or hardcoded keys violates security best practices and may result in account compromise. Use RAM users with least-privilege policies and rotate keys regularly.
</Warning>

## TLS requirements

| Requirement | Minimum | Recommended |
|-------------|---------|-------------|
| TLS version | TLS 1.2 | TLS 1.3 |
| SDK HTTP client | Must support TLS 1.2+ | Configure to prefer TLS 1.3 |
| Certificate validation | Must be enabled | Never disable in production |

All SDKs use HTTPS by default. Do not override the endpoint to use HTTP in production.

## Signature version

| Version | Status | SDKs |
|---------|--------|------|
| Signature V4 (HMAC-SHA256) | **Recommended** | All V2 SDKs use V4 by default |
| Signature V1 (HMAC-SHA1) | Supported | V1 SDKs; will be deprecated |

V2 SDKs default to Signature V4. If you are using a V1 SDK, consider enabling V4 signing where supported for improved security.

## Error handling

SDKs must handle the following error categories:

| Category | HTTP Status | Action |
|----------|-------------|--------|
| Client errors | 4xx | Fix request (do not retry) |
| Throttling | 429, 503 | Retry with exponential backoff |
| Server errors | 500, 502, 504 | Retry with exponential backoff |
| Network errors | -- | Retry with exponential backoff |

### Retry configuration

| Parameter | Default | Recommended |
|-----------|---------|-------------|
| Max retries | 3 | 3-5 |
| Base delay | 200ms | 200-500ms |
| Max delay | 30s | 20-60s |
| Backoff strategy | Exponential with jitter | Full jitter |

## Compliance checklist

- [ ] Using V2 SDK or latest V1 SDK version
- [ ] Authentication uses STS, instance roles, or RAM users (not root credentials)
- [ ] No hardcoded AccessKeys in source code
- [ ] TLS 1.2+ enforced; certificate validation enabled
- [ ] Signature V4 used where available
- [ ] Retry logic with exponential backoff for 5xx and 503 errors
- [ ] SDK client instances reused (connection pooling)
- [ ] SDK dependencies updated to latest patch versions
- [ ] Logging enabled for debugging without exposing credentials
