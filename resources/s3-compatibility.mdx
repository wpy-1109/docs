Alibaba Cloud OSS is compatible with the Amazon S3 API. You can use AWS SDKs, S3-compatible tools, and existing S3 applications to access OSS by configuring the endpoint and access credentials.

<Note>
OSS uses S3-compatible endpoints in the format `https://s3.oss-{region}.aliyuncs.com`. This is different from the standard OSS endpoint `https://oss-{region}.aliyuncs.com`. Use the S3-compatible endpoint when connecting with AWS SDKs.
</Note>

<Warning>
**China mainland custom domain policy (March 2025)**: New users who activated OSS after March 2025 must bind a custom domain name (CNAME) before performing data operations on buckets in Chinese mainland regions. This applies to both native OSS SDKs and S3-compatible access. See [Custom domain names](/guides/networking/custom-domain-names) for setup instructions.
</Warning>

## Quick start: connect with an AWS SDK

To access OSS with an AWS SDK, you need two things:

1. **S3-compatible endpoint**: `https://s3.oss-{region}.aliyuncs.com` (public) or `https://s3.oss-{region}-internal.aliyuncs.com` (internal/VPC)
2. **OSS AccessKey pair**: Create one in the [RAM console](https://ram.console.alibabacloud.com/users/create). AWS credentials and OSS credentials are separate systems and cannot be used interchangeably.

```python
import boto3
from botocore.config import Config

s3 = boto3.client(
    's3',
    endpoint_url='https://s3.oss-cn-hangzhou.aliyuncs.com',
    aws_access_key_id='LTAI5t...',
    aws_secret_access_key='yourAccessKeySecret',
    config=Config(
        signature_version='s3',
        s3={'addressing_style': 'virtual'}
    )
)

# List all buckets
for bucket in s3.list_buckets()['Buckets']:
    print(bucket['Name'])
```

### Client-side SDKs

<Warning>
Never hard-code permanent AccessKey credentials in client-side applications (browser, Android, iOS). Use [STS temporary credentials](/guides/access-control/sts-temporary-credentials) obtained from your server.
</Warning>

<Tabs>
  <Tab title="Browser">
```javascript Browser (AWS SDK v3)
import { S3Client } from '@aws-sdk/client-s3';

// Obtain STS temporary credentials from your server
async function getSTSCredentials() {
    const response = await fetch('https://your-server.com/api/sts-token');
    return await response.json();
}

const client = new S3Client({
    region: 'cn-hangzhou',
    endpoint: 'https://s3.oss-cn-hangzhou.aliyuncs.com',
    credentials: async () => {
        const creds = await getSTSCredentials();
        return {
            accessKeyId: creds.accessKeyId,
            secretAccessKey: creds.secretAccessKey,
            sessionToken: creds.securityToken,
            expiration: new Date(creds.expiration)
        };
    }
});
```

## Key differences from S3

### Endpoint format

OSS uses region-specific S3-compatible endpoints. Replace `{region}` with the region where your bucket is located, such as `cn-hangzhou` or `ap-southeast-1`.

| Endpoint Type | Format | Example |
|--------------|--------|---------|
| **S3-compatible public** | `https://s3.oss-{region}.aliyuncs.com` | `https://s3.oss-cn-hangzhou.aliyuncs.com` |
| **S3-compatible internal** | `https://s3.oss-{region}-internal.aliyuncs.com` | `https://s3.oss-cn-hangzhou-internal.aliyuncs.com` |
| **Standard OSS public** | `https://oss-{region}.aliyuncs.com` | `https://oss-cn-hangzhou.aliyuncs.com` |

For a complete list of regions, see [Regions and endpoints](/get-started/concepts/regions-and-endpoints).

### Request style (virtual-hosted only)

OSS supports **only virtual-hosted style** requests. Path-style requests are not supported. The bucket name must be a subdomain in the endpoint URL.

```
# Virtual-hosted style (supported by OSS)
https://examplebucket.oss-cn-hangzhou.aliyuncs.com/example.txt

# Path style (NOT supported by OSS)
https://s3.oss-cn-hangzhou.aliyuncs.com/examplebucket/example.txt
```

### ACL mapping

OSS supports only three ACL modes from S3: `private`, `public-read`, and `public-read-write`.

| Level | Permission | S3 Behavior | OSS Behavior |
|-------|-----------|-------------|--------------|
| Bucket | READ | List bucket contents | Read objects (if no object ACL is set) |
| Bucket | WRITE | Write or overwrite objects | Create or overwrite objects |
| Bucket | READ_ACP | Read bucket ACL | Owner and authorized RAM users only |
| Bucket | WRITE_ACP | Configure bucket ACL | Owner and authorized RAM users only |
| Object | READ | Read an object | Read an object |
| Object | WRITE | N/A | Overwrite an object |
| Object | READ_ACP | Read object ACL | Owner and authorized RAM users only |
| Object | WRITE_ACP | Configure object ACL | Owner and authorized RAM users only |

### Storage class mapping

| S3 Storage Class | OSS Storage Class | Notes |
|-----------------|-------------------|-------|
| `STANDARD` | Standard | Direct equivalent |
| `STANDARD_IA` | IA (Infrequent Access) | Direct equivalent |
| `GLACIER` | Archive | Restore required unless direct read is enabled |
| `DEEP_ARCHIVE` | Cold Archive / Deep Cold Archive | OSS has two tiers for deep archival |

OSS ignores the restoration duration configured in S3 API operations. By default, the restored state lasts one day and can be extended up to seven days.

### ETag differences

- **PUT uploads**: OSS returns ETags in **uppercase**; S3 returns ETags in **lowercase**. Configure your client for case-insensitive ETag comparison.
- **Multipart uploads**: OSS calculates ETag values using a different method than S3.

## Troubleshooting

<AccordionGroup>
  <Accordion title="InvalidArgument: aws-chunked encoding is not supported">
    **Symptom**: Upload fails with the error:

    ```
    InvalidArgument: aws-chunked encoding is not supported
    with the specified x-amz-content-sha256 value
    ```

    **Cause**: OSS supports AWS Signature V4 but does not support the `Transfer-Encoding: chunked` transfer method. Some AWS SDKs tie Signature V4 to chunked encoding by default.

    **Solution by SDK**:

    | SDK | Solution | Reason |
    |-----|----------|--------|
    | **Python (boto3)** | Use `signature_version='s3'` (V2) | boto3's V4 implementation is tightly coupled with chunked encoding |
    | **Java 1.x** | `.withChunkedEncodingDisabled(true)` | Chunked encoding can be disabled independently |
    | **Java 2.x** | `.chunkedEncodingEnabled(false)` | Chunked encoding can be disabled independently |
    | **Go v1/v2** | No action needed | Does not use chunked encoding by default |
    | **Node.js v3** | No action needed | Does not use chunked encoding by default |

    **Python fix example**:

    ```python
    # Incorrect: boto3 V4 uses chunked encoding (fails with OSS)
    s3 = boto3.client('s3',
        endpoint_url='https://s3.oss-cn-hangzhou.aliyuncs.com',
        config=Config(signature_version='v4'))

    # Correct: use V2 signature to avoid chunked encoding
    s3 = boto3.client('s3',
        endpoint_url='https://s3.oss-cn-hangzhou.aliyuncs.com',
        config=Config(signature_version='s3'))
    ```
  </Accordion>
  <Accordion title="SignatureDoesNotMatch">
    **Symptom**: All requests fail with `SignatureDoesNotMatch`.

    **Common causes and solutions**:

    1. **Wrong credentials**: You are using an AWS AccessKey instead of an OSS AccessKey. These are separate credential systems. Create an AccessKey pair in the [RAM console](https://ram.console.alibabacloud.com/users/create).

    2. **Clock drift**: The OSS server rejects requests with timestamps more than 15 minutes off. Check your server time with `date -u` and sync with NTP if needed.

    3. **Wrong endpoint**: The signature fails if the endpoint points to an AWS domain (e.g., `s3.amazonaws.com`) or the wrong OSS region. Use `https://s3.oss-{region}.aliyuncs.com` where the region matches your bucket.

    4. **Missing boto3 config**: If using Python boto3 without `Config(signature_version='s3')`, boto3 defaults to Signature V4, which causes failures.

    **Verify your configuration** with ossutil:

    ```bash
    ossutil ls oss://examplebucket \
      --access-key-id LTAI5t... \
      --access-key-secret yourAccessKeySecret \
      --endpoint oss-cn-hangzhou.aliyuncs.com
    ```

    If this command succeeds, your credentials and endpoint are correct -- the issue is in your SDK configuration.
  </Accordion>
  <Accordion title="NoSuchBucket or AccessDenied">
    **Symptom**: Requests fail with `NoSuchBucket` (404) or `AccessDenied` (403).

    **Common causes and solutions**:

    1. **Region mismatch**: Each OSS bucket belongs to a specific region. The endpoint must match the bucket's region exactly. OSS does not support cross-region access or automatic redirection like S3. If your bucket is in `cn-hangzhou`, you must use `s3.oss-cn-hangzhou.aliyuncs.com`.

    2. **Insufficient RAM permissions**: Verify that the RAM user has the necessary permissions (`oss:ListObjects`, `oss:GetObject`, `oss:PutObject`) for the target bucket.

    3. **Bucket naming with underscores**: When using virtual-hosted style, bucket names must comply with DNS conventions (no underscores). If your bucket name contains `_`, either use path-style in your SDK configuration or create a new bucket with a compliant name.
  </Accordion>
</AccordionGroup>

## Performance optimization

AWS SDKs provide transfer acceleration features that work with OSS. These mechanisms use the S3 Multipart Upload API, which OSS fully supports.

| SDK | Feature | Description |
|-----|---------|-------------|
| **Python (boto3)** | `TransferConfig` | Configure `multipart_threshold`, `max_concurrency`, and `multipart_chunksize` for automatic concurrent multipart uploads |
| **Java** | `TransferManager` | Encapsulates multipart upload, concurrent transfers, and automatic retries. Selects optimal transfer strategy based on file size |
| **Go** | `s3manager.Uploader` | Built-in concurrent multipart upload with automatic part splitting and retry logic |
| **Node.js** | `Upload` from `@aws-sdk/lib-storage` | Supports streaming uploads to reduce memory usage for large files |

## Migration considerations

When migrating from S3 to OSS:

1. **Update endpoints** -- Replace S3 endpoints with `https://s3.oss-{region}.aliyuncs.com`
2. **Switch to virtual-hosted style** -- OSS does not support path-style requests
3. **Update ACL configurations** -- OSS supports only `private`, `public-read`, and `public-read-write`
4. **Handle ETag differences** -- Use case-insensitive ETag comparison
5. **Map storage classes** -- Update any storage class references to OSS equivalents
6. **Review unsupported operations** -- Verify your application does not rely on S3 operations not listed in the compatibility tables above
7. **Configure signature version** -- Use Signature V2 for Python boto3; disable chunked encoding for Java SDKs

For a comprehensive migration guide, see [Migrate from S3 to OSS](/resources/best-practices/migration-from-s3).
