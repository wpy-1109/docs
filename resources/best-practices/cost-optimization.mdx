OSS costs are driven by three dimensions: **storage capacity**, **data transfer (traffic)**, and **API requests**. By optimizing each dimension, you can significantly reduce your total OSS spending.

## Cost model overview

Understanding the OSS cost model is the first step to optimization.

| Cost Dimension | What Drives It | Optimization Strategy |
|---------------|----------------|----------------------|
| **Storage** | Volume of data x storage class unit price | Use cheaper storage classes for infrequently accessed data |
| **Traffic** | Outbound internet data transfer | Use internal endpoints, CDN, or pay-by-requester |
| **Requests** | Number of API calls (PUT, GET, LIST, etc.) | Batch operations, cache metadata locally, use ListObjectsV2 |

## Storage class selection

Choose the right storage class based on how frequently your data is accessed. Moving data to a cheaper tier is the single most effective way to reduce costs.

| Storage Class | Best For | Relative Storage Cost |
|--------------|----------|----------------------|
| **Standard** | Data accessed daily | Highest |
| **Infrequent Access (IA)** | Data accessed < once/month | ~50% lower than Standard |
| **Archive** | Data accessed < once/quarter | ~70% lower than Standard |
| **Cold Archive** | Long-term archival | ~85% lower than Standard |
| **Deep Cold Archive** | Compliance/regulatory | ~90% lower than Standard |

<Warning>
If you delete or transition objects before the minimum storage duration (30 days for IA, 60 days for Archive, 180 days for Cold Archive/Deep Cold Archive), you are still charged for the full minimum period. Factor this into your cost calculations.
</Warning>

For a detailed comparison, see [Storage classes](/get-started/concepts/storage-classes).

## Lifecycle rules for automatic tiering

Lifecycle rules automatically transition objects to cheaper storage classes or delete expired data. This is the most effective hands-off cost optimization.

### Transition rules by access frequency

Configure transition rules based on how your data access patterns change over time.

```
Rule: "archive-old-logs"
Scope: Prefix = "logs/"
├── Transition to IA: 30 days
├── Transition to Archive: 90 days
├── Transition to Cold Archive: 365 days
├── Delete: 2555 days (7 years)
└── Abort incomplete multipart uploads: 7 days
```

### Clean up previous versions

When versioning is enabled, objects you overwrite or delete are saved as previous versions. These accumulate and increase storage costs. Configure lifecycle rules to automatically delete old versions after a specified number of days.

For the specific configuration steps, see [Lifecycle rules](/guides/data-management/lifecycle-rules).

### Abort incomplete multipart uploads

Failed or abandoned multipart uploads consume storage but serve no purpose. Add a lifecycle rule to clean them up automatically.

```xml
<Rule>
  <ID>abort-multipart</ID>
  <Prefix></Prefix>
  <Status>Enabled</Status>
  <AbortMultipartUpload>
    <Days>7</Days>
  </AbortMultipartUpload>
</Rule>
```

<Tip>
Apply this rule to every bucket. A 7-day cleanup window catches abandoned uploads while allowing time for long-running legitimate uploads to complete.
</Tip>

## Reduce traffic costs

### Use internal endpoints

Access OSS from ECS instances in the same region using internal endpoints. Internal traffic is free.

```
# Internet endpoint (charges traffic fees)
https://examplebucket.oss-cn-hangzhou.aliyuncs.com

# Internal endpoint (free traffic within the same region)
https://examplebucket.oss-cn-hangzhou-internal.aliyuncs.com
```

### CDN acceleration

Route end-user downloads through Alibaba Cloud CDN. OSS-to-CDN back-to-origin traffic costs less than direct internet egress.

<Steps>
  <Step title="Create a CDN distribution">
    Point the CDN distribution origin to your OSS bucket endpoint.
  </Step>
  <Step title="Configure caching">
    Set `Cache-Control` headers on your objects for effective edge caching.
  </Step>
  <Step title="Update client URLs">
    Redirect end-user download requests to the CDN domain instead of the OSS endpoint.
  </Step>
</Steps>

For details, see [CDN acceleration](/guides/networking/cdn-acceleration).

### Pay-by-requester

For shared data scenarios where external users access your bucket, enable pay-by-requester mode. The requester pays for outbound traffic and request fees instead of the bucket owner.

```xml
<RequestPaymentConfiguration>
  <Payer>Requester</Payer>
</RequestPaymentConfiguration>
```

For configuration details, see [Pay-by-requester](/guides/buckets/pay-by-requester).

<Note>
Outbound data transfer resource plans cannot offset traffic fees incurred by pay-by-requester buckets. If you use both features, the requester is charged at pay-as-you-go rates for traffic regardless of any data transfer plans you have purchased.
</Note>

## Optimize request costs

- **Batch small files**: Combine multiple small operations to reduce the total number of API calls.
- **Use ListObjectsV2**: More efficient than ListObjects for large result sets.
- **Cache metadata locally**: Avoid unnecessary HEAD requests by caching object metadata on the client side.
- **Use OSS Select**: Query data in-place with SQL expressions instead of downloading entire files. See [Select Object](/guides/objects/select-object).
- **Upload as Standard, transition via lifecycle**: Uploading objects directly as Deep Cold Archive incurs higher PUT request fees. Upload as Standard and let lifecycle rules transition them automatically.

## Purchase resource plans

Prepaid resource plans (storage capacity units, or SCUs) offer significant discounts compared to pay-as-you-go pricing.

| Plan Type | Discount | Best For |
|-----------|----------|----------|
| **Storage capacity units (SCU)** | Up to 50%+ off storage | Predictable storage workloads |
| **Data transfer plans** | Discounted outbound traffic | High-bandwidth applications |

SCUs can offset Standard, IA, and Archive storage fees. They are available in 1-month, 6-month, and 1-year terms. For details, see [Resource plans](/resources/pricing/resource-plans).

## Monitor and control costs

<Steps>
  <Step title="Track daily costs">
    Use the Billing Management console to monitor daily OSS spending broken down by storage, traffic, and requests.
  </Step>
  <Step title="Set budget alerts">
    Configure billing alerts to notify you when spending exceeds thresholds. This catches unexpected cost spikes early.
  </Step>
  <Step title="Tag buckets for cost allocation">
    Use bucket tags to attribute costs to teams, projects, or environments.
  </Step>
  <Step title="Review storage usage">
    Query bucket-level usage regularly with `GetBucketStat` to identify unexpected capacity growth. If storage grows unexpectedly, check for unauthorized access and tighten bucket ACLs or policies.

    ```bash
    # Check bucket storage statistics
    ossutil stat oss://examplebucket
    ```
  </Step>
</Steps>

## Cost optimization checklist

Use this checklist to audit your OSS cost efficiency:

- [ ] **Internal endpoints**: All ECS-to-OSS traffic uses `-internal` endpoints
- [ ] **Lifecycle rules on log buckets**: Logs auto-transition to IA after 30 days, Archive after 90 days
- [ ] **Abort multipart uploads**: Lifecycle rule cleans up incomplete uploads in every bucket
- [ ] **Previous version cleanup**: Lifecycle rule removes old versions in versioned buckets
- [ ] **CDN for downloads**: End-user downloads route through CDN, not direct internet egress
- [ ] **Storage class audit**: No frequently accessed data stored in Archive; no rarely accessed data in Standard
- [ ] **Resource plans**: SCUs purchased for predictable storage workloads
- [ ] **Budget alerts**: Billing alerts configured for unexpected spending
