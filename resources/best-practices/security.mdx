## Access control

### Keep buckets private

Set bucket and object ACLs to `private` unless you have a specific need for public access:

- `public-read-write`: Anyone can read and write -- risk of data leakage, cost surges, and illegal content
- `public-read`: Anyone can read -- risk of data exposure and bandwidth costs
- `private`: Only the owner and authorized users have access (recommended default)

### Use RAM users

Never use your primary Alibaba Cloud account AccessKey for API access. Create dedicated RAM users with minimum required permissions.

```json
{
  "Version": "1",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "oss:GetObject",
        "oss:PutObject"
      ],
      "Resource": [
        "acs:oss:*:*:my-bucket/uploads/*"
      ]
    }
  ]
}
```

### Never hardcode accessKeys

- **Server-side**: Use the Alibaba Cloud SDK Managed Credentials Plugin or instance roles
- **Client-side**: Never embed AccessKeys in client code; use STS temporary credentials instead
- **CI/CD**: Use environment variables or secrets management

### Use STS temporary credentials

Issue time-limited, scoped credentials for client applications:

1. Client requests a token from your server
2. Your server calls STS `AssumeRole` with a scoped policy
3. Client uses the temporary credentials to access OSS directly
4. Credentials expire automatically

### Enable block public access

Prevent accidental public exposure at the account or bucket level:

```xml
<PublicAccessBlockConfiguration>
  <BlockPublicAccess>true</BlockPublicAccess>
</PublicAccessBlockConfiguration>
```

### Configure bucket policies

Use bucket policies for fine-grained access control:

- IP-based restrictions
- VPC endpoint restrictions
- Require HTTPS
- Time-based access windows

## Encryption

| Method | Description | Key Management |
|--------|-------------|----------------|
| **SSE-KMS** | Server-side encryption with KMS | Managed by KMS with CMK rotation |
| **SSE-OSS (AES256)** | Server-side encryption managed by OSS | Fully managed, no configuration needed |
| **Client-Side** | Encrypt before upload | You manage encryption keys |
| **TLS/HTTPS** | Encryption in transit | Enforce via bucket policy |

### Enforce HTTPS

Create a bucket policy that denies non-HTTPS requests:

```json
{
  "Version": "1",
  "Statement": [
    {
      "Effect": "Deny",
      "Action": "oss:*",
      "Principal": ["*"],
      "Resource": ["acs:oss:*:*:my-bucket/*"],
      "Condition": {
        "Bool": {
          "acs:SecureTransport": "false"
        }
      }
    }
  ]
}
```

### Set minimum TLS version

Enforce TLS 1.2 or higher:

```xml
<HttpsConfiguration>
  <TLS>
    <Enable>true</Enable>
    <TLSVersion>TLSv1.2</TLSVersion>
  </TLS>
</HttpsConfiguration>
```

## Data protection

### Enable versioning

Protect against accidental deletion and overwrites:

- Deleted objects get a delete marker (recoverable)
- Overwritten objects retain previous versions
- Combine with lifecycle rules to manage version retention

### Use WORM retention

For regulatory compliance, enable Write Once Read Many (WORM):

- Objects cannot be deleted or modified during the retention period
- Lock the policy to make it irreversible
- Supports SEC, FINRA, and other compliance requirements

### Cross-Region replication

Set up disaster recovery with CRR:

- Real-time replication to a secondary region
- Enable RTC for guaranteed replication within 10 minutes
- Supports both new and historical object replication

## Monitoring and auditing

- **Access Logging**: Enable access logs to a separate bucket
- **ActionTrail**: API-level auditing for compliance
- **Cloud Monitor**: Set alarms for unusual patterns (high error rates, unexpected traffic)
- **Hotlink Protection**: Configure Referer whitelist to prevent bandwidth theft

## Enable MFA

Enable multi-factor authentication for:

- Alibaba Cloud account login
- RAM user console access
- Critical operations (bucket deletion, policy changes)

## Security checklist

- [ ] All buckets set to `private` ACL
- [ ] Block Public Access enabled
- [ ] AccessKeys not hardcoded in code
- [ ] RAM users with least-privilege policies
- [ ] STS used for client-side access
- [ ] Server-side encryption enabled
- [ ] HTTPS enforced via bucket policy
- [ ] Versioning enabled on critical buckets
- [ ] Access logging enabled
- [ ] MFA enabled for administrative accounts
