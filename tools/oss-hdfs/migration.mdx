## Migration approaches

| Approach | Best For | Tool |
|----------|----------|------|
| **Jindo DistCp** | Large-scale parallel copy | JindoSDK |
| **Hadoop DistCp** | Standard Hadoop migration | Built-in Hadoop |
| **ossutil** | Simple file transfers | ossutil CLI |

## Using jindo distCp

Jindo DistCp is optimized for OSS and provides better performance than standard Hadoop DistCp.

```bash
hadoop jar jindo-distcp-x.x.x.jar \
  --src hdfs://namenode:8020/data/ \
  --dest oss://target-bucket/data/ \
  --parallelism 20 \
  --bandWidth 500
```

### Key options

| Option | Description |
|--------|-------------|
| `--src` | Source path (HDFS or OSS) |
| `--dest` | Destination path (OSS or HDFS) |
| `--parallelism` | Number of parallel copy tasks |
| `--bandWidth` | Bandwidth limit per task (MB/s) |
| `--srcPattern` | Regex filter for source files |
| `--update` | Only copy new or changed files |

## Using hadoop distCp

```bash
hadoop distcp \
  -Dfs.oss.endpoint=cn-hangzhou.oss-dls.aliyuncs.com \
  hdfs://namenode:8020/data/ \
  oss://target-bucket/data/
```

## Migration best practices

1. **Validate before migration**: Run a dry-run to estimate data volume and time
2. **Use incremental sync**: Migrate in phases, then switch over
3. **Verify data integrity**: Compare file counts and checksums after migration
4. **Update applications**: Change paths from `hdfs://` to `oss://` in configurations
5. **Test thoroughly**: Run end-to-end tests with your compute workloads before full cutover
6. **Monitor performance**: Compare query execution times between HDFS and OSS-HDFS
